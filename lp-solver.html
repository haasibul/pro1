Overview
What this project demonstrates

This project came from a class activity asking students to build a general LP solver for maximization problems using AI assistance, then host it publicly on GitHub Pages. The tool had to work for any LP — not just pre-built examples.

This page IS the project. The solver below implements the Simplex Method: it converts constraints to augmented form by adding slack variables, then iterates pivot operations until all objective-row coefficients are non-negative — the optimality condition.

Building a working solver was the best way to understand the algorithm. Writing the pivot loop line by line forced me to understand what every step actually does mathematically — not just memorize the rule.

Build Process — Five Steps
01
Understand the requirements. The solver must accept arbitrary inputs, build augmented form, and apply the entering/leaving variable rules from the textbook.
02
Write a precise AI prompt. Vague prompts produce black-box code. The prompt specified: -c[j] in objective row, min-ratio test, row-reduce after each pivot.
03
Generate and review. AI produced the simplex loop. I verified it against in-class examples (Street Food Co., Pinocchio) by hand before trusting any output.
04
Debug a critical bug. Using +c[j] instead of -c[j] produced a wrong-but-plausible answer. Only hand-tracing caught it.
05
Deploy to GitHub Pages. Push index.html → Settings → Pages. Live in 60 seconds. Zero server, zero cost.

Simplex Method — Core Concepts

Augmented Form
Slack Variables
Each ≤ constraint becomes an equality by adding a slack variable sᵢ ≥ 0. The initial basis is all slacks, mapping to the corner at the origin (x₁=x₂=0).
Entering Variable
Most-Negative Rule
The entering variable has the most negative coefficient in the objective row. It's the variable that increases Z fastest per unit added to the basis.
Leaving Variable
Min-Ratio Test
For each row with a positive pivot column entry, compute RHS/coefficient. The smallest ratio leaves — preventing any variable from becoming negative.
Optimality Condition
All Coefficients ≥ 0
The algorithm halts when every objective-row coefficient is non-negative. No variable can improve Z — we are at the optimal corner point of the feasible region.

function pivot(T, pR, pC) {
  const pv = T[pR][pC];
  T[pR] = T[pR].map(v => v / pv);
  for (let i=0; i<T.length; i++) {
    if (i===pR) continue;
    const f = T[i][pC];
    T[i] = T[i].map((v,j) =>
      v - f * T[pR][j]);
  }
}

AI Debugging Conversation
Student Prompt
Build a general LP maximization solver in vanilla JS. It should: take objective coefficients and an A-matrix/b-vector, add slack variables to build augmented form, iterate simplex using the most-negative-coefficient entering rule and min-ratio-test leaving rule, and return the optimal solution with full tableau history.
Assistant
The key is representing the tableau as a 2D array where the last row is the objective and the last column is the RHS. Set up the objective row with negated coefficients (-c[j])…
Student — Bug Report
The solver returns Z=30 for the street food problem but the correct answer is Z=38. It doesn't crash — it just stops early. What went wrong?
Assistant — Fix
The issue is in the objective row setup. You stored +c[j] instead of -c[j]. The optimality check reads "all obj-row ≥ 0" — with positive values the algorithm thinks it's already optimal at a sub-optimal corner. Change to objRow[j] = -c[j].
Reflection
What Did I Learn?
Knowledge

Concepts

  • Slack variables turn each ≤ into an equality, making the tableau square for pivoting
  • The objective row stores negated coefficients — Z accumulates through row operations
  • Every simplex iteration is a move from one corner to an adjacent, strictly better one
  • The min-ratio test is what prevents basic variables from going negative (infeasible)
  • The algorithm terminates in finite steps because each pivot strictly improves Z
  • Binding constraints at the optimum define the active edges meeting at that vertex
Skills

Capabilities

  • Writing algorithmic prompts with enough precision to get correct numerical code
  • Debugging by tracing a known example (street food) through the tableau by hand
  • Implementing canvas-based 2D feasible region and constraint line visualization
  • Deploying a static site to GitHub Pages in under 5 minutes
  • Reading and verifying a simplex tableau: spotting the identity submatrix of the basis
  • Separating solver logic, UI, and drawing into clean, independent functions
Reflection

Surprises

The most revealing moment was the sign error bug. Using +c instead of -c in the objective row produced a wrong answer that looked completely reasonable — the solver didn't crash, it just stopped one iteration too early. This taught me that AI-generated numerical code requires verification against a hand-calculated example every single time.


Building the tool also made the algorithm feel physical. Reading about "pivot operations" is abstract; writing the row-reduction loop made me understand exactly what each pivot does to the tableau and why it corresponds to moving between corner points.


Real-world solvers like CPLEX and Gurobi use this exact logic — extended with the revised simplex, dual simplex, and branch & bound. The pivot rule I implemented is unchanged from Dantzig's original 1947 paper.
