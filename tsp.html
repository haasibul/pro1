Section 1 — Project Overview

What this page is about

This page documents my Week 12 Networks project on the Travelling Salesman Problem (TSP) — a classic optimization challenge. The assignment came from a delivery-business scenario: a driver must depart from a warehouse (WH), deliver packages to four towns (A, B, C, D), visit each exactly once, and return to WH — all while minimizing total distance (and therefore fuel cost).

The core concept being demonstrated is combinatorial optimization through brute-force enumeration: since there are only 5 nodes, all 24 possible round-trip routes can be listed, their costs calculated, and the minimum identified directly. I also compare this against a greedy "nearest neighbour" heuristic, which is faster but not always optimal.

This matters because real logistics networks are exactly this problem at scale — except with hundreds or thousands of stops, brute force becomes impossible (factorial growth), so understanding where the heuristic agrees with — or diverges from — the true optimum is the actual lesson. The project also explores a constrained variant: what happens when one road (C ↔ D) becomes unavailable.

Quick facts

• 5 nodes total (WH + 4 towns)
• 24 possible closed routes (4! permutations)
• Optimal cost, all roads open: 25
• Optimal cost, C↔D blocked: 30
• 4 distinct routes tie for the optimum of 25
• Greedy heuristic happens to match the optimum here

      Shortcut Notes & Strategy
      Shortcut tricks — choose smallest value at each step:
      1 Start with WH → A(6) [smallest of 6,8,7,10]
      2 From A — visited: B(4), C(3), D(7) → pick C(3)
      3 From C — visited: B(5), D(2) → pick D, but only one option remains
      4 From D — visited: B(6) → only option
      5 Return WH from B
      This is exactly the nearest-neighbour greedy heuristic implemented and visualized in Section 2 — and in this case it lands on the true optimum (25).
      Section 4 — What Did I Learn?

      Reflection

      🧠 Knowledge

      I learned what the Travelling Salesman Problem actually is: finding the minimum-cost Hamiltonian cycle through a weighted graph. With only 4 towns plus a warehouse, there are (n−1)!/2 = 12 structurally unique routes (24 if direction is counted separately), which is small enough to brute-force by hand. I also learned the difference between an exact algorithm (brute force — guaranteed optimal, but factorial time) and a heuristic (nearest neighbour — fast, but only sometimes optimal).

      🛠️ Skills

      Practiced systematic enumeration and arithmetic checking by hand before automating it. Built a distance matrix from a network diagram and back again. Practiced optimization thinking — comparing multiple candidate solutions against a baseline rather than trusting the first answer found. Also practiced representing the same network in two different visual forms (star graph vs. complete graph) to check my own distance values for consistency.

      💡 Reflection

      The hardest part was keeping track of which routes I'd already tried by hand — duplicates and reversed routes (e.g. WH→A→C→D→B→WH vs. WH→B→D→C→A→WH) have the same cost since the graph is symmetric, which was surprising until I realized a route and its reverse are always identical in cost. In the real world, this scale of problem is what delivery companies like UPS or FedEx solve constantly — except with thousands of stops, where brute force is impossible and heuristics (or more advanced methods like genetic algorithms or branch-and-bound) become necessary. The constrained scenario (Part 2) also showed how a single road closure can meaningfully raise total cost — from 25 to 30, a 20% increase — which mirrors real supply-chain disruptions.
